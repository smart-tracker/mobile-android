## 🔴 Блокеры публичного релиза

### BR-1. Убрать `debug_code` из ответа `POST /auth/register`
**Что:** сейчас код email-верификации возвращается открытым текстом в теле ответа.
**Зачем:** любой, кто видит ответ (перехват трафика, логи), регистрирует аккаунт
на чужой email без доступа к почте. Дыра безопасности, не фича.
**Как:** отдавать только при `DEBUG=true` в конфиге сервера, в production — никогда.
**Android:** поле намеренно не включено в `RegisterResultDto` — клиентских правок не нужно.
**Приёмка:** production-ответ `/auth/register` не содержит `debug_code`.

### BR-2. Зеркальная проверка почтового домена в `POST /auth/register` (149-ФЗ)
**Что:** отклонять регистрацию с email не из списка российских почтовых доменов —
HTTP 400 с понятным `detail` (например, `"Registration requires a Russian email provider"`).
**Зачем:** 149-ФЗ (поправки от 01.12.2023) — регистрация только через российские
почты/номера/ЕСИА/российские ID. Клиентская проверка уже есть, но обходится
прямым HTTP-запросом — юридически прикрывает только серверная.
**Важно:** проверка ТОЛЬКО на регистрации. `/auth/login` и `/password-reset/*`
не ограничивать — существующие аккаунты с иностранной почтой должны работать.
**Android:** готово — `EmailValidator.isAllowedDomain` + `AllowedEmailDomainsRepository`,
список: Яндекс (`yandex.ru`, `ya.ru`, `yandex.com`, `narod.ru`),
VK/Mail.ru (`mail.ru`, `bk.ru`, `list.ru`, `inbox.ru`, `internet.ru`, `vk.com`),
Rambler (`rambler.ru`, `lenta.ru`, `autorambler.ru`, `myrambler.ru`, `ro.ru`).
Списки клиента и сервера должны совпадать.
**Приёмка:** `curl POST /auth/register` с `user@gmail.com` → 400; с `user@yandex.ru` → 200.

### BR-3. Хеширование паролей — подтвердить bcrypt
**Что:** проверить, что пароли хешируются `bcrypt` (или argon2), а не `sha256_crypt`.
**Зачем:** быстрые хеши брутфорсятся на GPU; при утечке БД все пароли скомпрометированы.
**Приёмка:** хеши в таблице `users` начинаются с `$2b$` (bcrypt) / `$argon2`.

## 🟡 Высокий приоритет

### BR-4. `GET /auth/allowed-email-domains` — серверный список доменов
**Что:** новый публичный эндпоинт: `{"domains": ["yandex.ru", "mail.ru", ...]}`.
**Зачем:** сейчас список захардкожен в APK — любое изменение (новый провайдер,
корпоративные домены РФ, вузовские `@petrsu.ru`) требует релиза приложения.
**Android:** инфраструктура готова — заменить `AllowedEmailDomainsRepositoryImpl`
на сетевую реализацию с кэшем (план в TODO-комментарии внутри impl),
хардкод останется fallback'ом при недоступности сети.
**Приёмка:** эндпоинт отдаёт актуальный список; изменение списка на сервере
подхватывается клиентом без обновления APK.

### BR-5. `gps_track` с временны́ми метками в `GET /training/{id}/get_training`
**Что:** заменить GeoJSON LineString без времени:
```json
{"type": "LineString", "coordinates": [[lon, lat, alt], ...]}
```
на массив точек с `recorded_at`:
```json
"gps_track": [
  {"lat": 61.774, "lon": 34.379, "alt": 5, "recorded_at": 1716890640613},
  ...
]
```
**Зачем:** без временных меток невозможно показать elapsed и скорость
в scrub-оверлее истории тренировок; также блокирует экспорт GPX.
**Android после изменения:** обновить `GetTrainingDetailResponseDto`
(убрать `JsonElement?`, вернуть `List<GpsTrackPointDto>?` + `recorded_at`), маппер.
**Приёмка:** ответ содержит `recorded_at` (epoch millis UTC) для каждой точки.

### BR-6. Синхронизировать SQLAlchemy-модели с реальной схемой БД
**Что:** после ручного `ALTER TABLE users ALTER COLUMN last_name/middle_name/weight/height
DROP NOT NULL` модели на бэке не обновлены — схема в коде врёт.
**Зачем:** следующая миграция/деплой с autogenerate вернёт NOT NULL и сломает
регистрацию без необязательных полей (уже ловили HTTP 500 `NotNullViolationError`).
**Приёмка:** модели соответствуют БД; alembic-миграция зафиксирована в репо.

### BR-7. `POST /training/{id}/save_training` — 500 при отсутствии полей
**Что:** тело без `total_distance_meters`/`total_kilocalories` (например,
`{"time_end": "..."}`) вызывает HTTP 500. Должно быть 200 с дефолтами
или 422 с описанием.
**Зачем:** Gson на клиенте дропает null-поля; Android сейчас вынужден слать `0.0`
явно (воркэраунд) — сервер не должен падать от валидного частичного тела.
**Приёмка:** запрос без опциональных полей → не 500.

### BR-14. Подтвердить идемпотентность `POST /training/{id}/gps_points` по `batch_id`
**Что:** проверить (и при отсутствии — реализовать), что повторная отправка батча
с тем же `batch_id` не создаёт дубликаты GPS-точек, а возвращает успех.
**Зачем:** Android ретраит неподтверждённые батчи с тем же `batch_id`
(sync-цикл каждые 10 сек + `SyncGpsPointsWorker` при офлайн-финише).
Сценарий: сервер сохранил точки, но ответ потерялся (таймаут/обрыв) →
клиент не получил подтверждение → ретрай того же батча. Без серверной
дедупликации каждая такая ситуация удваивает точки трека — трек в истории
«дрожит» и дистанция на сервере расходится с клиентской.
**Android:** уже реализовано в расчёте на дедупликацию: батч получает
`batch_id` (UUID) один раз и ретраится только с ним (`WHERE batchId IS NULL`
в DAO защищает от переприсвоения).
**Приёмка:** двойной `POST .../gps_points` с одинаковым `batch_id` → вторая
попытка не увеличивает число точек тренировки (идемпотентный 200/409).

### BR-15. Публичный URL политики конфиденциальности
**Что:** опубликовать политику конфиденциальности по публичному URL
(например, `https://runtastic.gottland.ru/privacy`) — обычная HTML-страница,
доступная без авторизации.
**Зачем:** RuStore (и Google Play) требуют ссылку на политику в карточке
приложения — без неё публикация невозможна. Также 152-ФЗ требует
опубликованного документа о политике обработки ПДн (ст. 18.1).
**Android:** черновик текста — в `LegalScreens.kt` (`PrivacyPolicyScreen`),
структурирован по 152-ФЗ. Перед публикацией: заполнить реквизиты оператора
(плейсхолдеры в тексте), юр-проверка. Тексты в приложении и по URL должны
совпадать.
**Приёмка:** URL открывается без авторизации, отдаёт актуальный текст политики;
ссылка принята кабинетом RuStore.

## 🟢 Средний / низкий приоритет

### BR-8. Инвалидация старых кодов верификации
При повторной отправке кода старые должны инвалидироваться (DELETE или
`SET verified_at`) — иначе окно перебора расширяется на все выданные коды.

### BR-9. Коды верификации через `secrets`, не `random`
`random` предсказуем — коды подбираются. `secrets.randbelow` / `secrets.choice`.

### BR-10. CORS — проверить конфигурацию
Не должно быть `allow_origins=["*"]` вместе с `allow_credentials=True`.

### BR-11. SMTP не должен блокировать event loop
Отправка email 3–10 сек синхронно = блокировка всех запросов FastAPI.
Вынести в background task / очередь.

### BR-12. Timezone — единообразие UTC
Убедиться, что `expires_at`/`created_at` сравниваются в UTC на всех путях
(регистрация, resend, verify) — рассинхрон даёт ложные «код истёк».

### BR-13. Uvicorn под systemd (ops)
Сейчас перезапуск только вручную от пользователя сервера. Падение процесса =
простой до ручного вмешательства. Нужен systemd-unit с `Restart=always`.
